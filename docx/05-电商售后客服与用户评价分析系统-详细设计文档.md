## 词汇表

| 术语                                  | 全称/说明                                        |
| ----------------------------------- | -------------------------------------------- |
| 用户（Consumer）                        | 通过微信小程序/H5 访问系统的消费者                          |
| 客服人员（Agent）                         | 商家客服或客服主管，使用 Web 管理后台处理咨询和工单                 |
| 平台管理员（Admin）                        | 负责系统全局配置、知识库、规则及数据分析的角色                      |
| 售后工单（Aftersale Ticket）              | 用户提交退换货、质量投诉等自动生成的业务单据                       |
| 意图识别（Intent Recognition）            | 基于大模型对用户咨询文本进行分类，得到意图类别及置信度                  |
| RAG（Retrieval-Augmented Generation） | 结合检索与生成：从知识库检索相关片段，拼接 Prompt 送入 LLM 生成回复     |
| 会话沉淀（Session Summarization）         | 每轮对话结束后自动生成 Markdown 格式的会话记录文档               |
| Agent 自进化（Evolving Agent）           | 基于会话沉淀文档自动归纳高频问题、优化回复话术，持续提升 AI 能力           |
| L1/L2/L3 级处理                        | 智能客服分级处理机制：L1 AI 全自动回复，L2 AI 辅助人工回复，L3 全人工处理 |
| 对策库（Solution Library）               | 基于历史评价和工单自动生成的标准处理方案与话术库                     |
| 公开评价爬虫（Public Comment Crawler）      | 按规定频率从淘宝/天猫等平台抓取商品公开评价，清洗后入库                 |

## 总体结构

系统采用五层微服务架构：

1. **接入层**：CDN + Nginx 负责静态资源分发、SSL 终结、负载均衡。  
2. **用户端层**：分别提供 H5/微信小程序（消费者入口）、React/Web 项目（商家客服入口）、React/Web 项目（平台管理员入口）。  
3. **业务服务层**：基于 FastAPI（或 Spring Boot）实现的无状态后端服务，提供 RESTful API 与 WebSocket 实时推送。  
4. **AI 引擎层**：独立 FastAPI 子应用（或独立服务），封装意图识别、多轮对话、RAG 回复生成、情感分析、主题归类、工单自动分类等核心 AI 能力。  
5. **数据层**：内部存储与检索系统，包括 MySQL（业务数据）、Redis（缓存+消息队列 Broker）、Elasticsearch（评价全文检索及日志）、Milvus（向量检索用于 RAG）、MinIO/OSS（对象存储，保存凭证图片、会话日志、MD 沉淀）。

### 5. 异步任务服务（Celery Worker）

| 任务名称            | 调用触发条件                                                     | 主要处理函数                                                                                                                                                     | 备注                         |
| --------------- | ---------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------- |
| 评价情感批量分析        | 每天凌晨 02:00 由 Celery Beat 触发，或手动通过接口触发                      | `batch_sentiment_analysis()`：从 MySQL 读取未分析的评价（`evaluation` 表中 `sentiment_analyzed = 0`），调用 `SentimentAnalyzer` 批量写回结果，更新标记                                 | 随机取 1 万条为一批，使用 Redis 锁防止重复 |
| 公开评价爬虫数据清洗 & 入库 | 爬虫任务完成后回调（或独立调度）                                           | `clean_and_store_crawler_data(raw_id)`：读取原始抓取数据（`crawler_raw` 表），去重（基于 `contentHash`），文本清洗（去除 HTML 标签、表情、敏感词），调用情感分析&主题归类，写入 `evaluation` 表                | 单日采集上限 50 万条               |
| 会话沉淀 MD 生成      | 每次会话结束后由业务层通过接口触发（`POST /ai/session/summarize`）或定时检测未生成的会话 | `generate_session_doc(sessionId)`：从 Redis 拉取完整聊天记录，调用 `SessionSummarizer` 生成 MD，上传至 OSS，将文档 URL 写回 `session_doc` 表                                         | 若生成失败会重试 3 次               |
| 定时统计报表生成        | 每小时或每天凌晨（依据业务需求）                                           | `generate_periodic_report(granularity)`：基于 MySQL 聚合工单、评价、咨询等数据，生成 JSON/Excel 并上传至 OSS，记录于 `report` 表                                                       | 支持按天/周/月粒度                 |
| Agent 自进化模型更新   | 每周日凌晨 03:00 自动触发，或手动通过管理后台触发                               | `run_evolution_compute()`：读取最近 7 天的 `session_doc`，统计高频意图、低置信度案例、满意度趋势，自动更新意图识别阈值表、`intent_threshold`，生成新对策草稿写入 `solution_lib` 表，同时写更新日志到 `evolution_log` 表 | 产出报告供管理后台查看                |
| 可选：异常告警检测       | 每 5 分钟检查一次                                                 | 检查 Redis 队列积压、DB 慢查询、服务错误率，发送告警至企业微信/钉钉                                                                                                                    | 通过监控体系实现                   |

### 8. 安全设计

1. **传输安全**  
   
   - 外部所有接口强制 HTTPS，TLS 版本 ≥ 1.2。  
   - 内部服务间通信亦使用 HTTPS（或在同一 VPC 内部使用 mTLS）。  

2. **身份鉴权与授权**  
   
   - 统一使用 JWT（HS256）进行身份认证，Access Token 有效期 15 分钟，Refresh Token 有效期 7 天，存放在前端 `localStorage`（或 httpOnly Cookie）。  
   - 各接口在网关层进行统一鉴权；后端再根据 JWT 中的角色（`user`、`agent`、`admin`）做细粒度权限校验（RBAC）。  
   - 敏感操作（如工单处理、规则修改、数据导出）需额外验证码或短信验证。  

3. **数据安全**  
   
   - 用户手机号、订单号、姓名等敏感字段在库中采用 AES-256 加密存储（列级加密），备份文件同样加密。  
   - 日志中对手机号、订单号进行脱敏（仅保留后四位）。  
   - 文件上传只允许 `jpg、jpeg、png、pdf`，单文件大小 ≤ 5 MB，后端调用 ClamAV 或类似病毒扫描。  

4. **防注入与防脚本攻击**  
   
   - 所有输入均采用参数化查询（ORM）或严格过滤，防止 SQL 注入。  
   - 前端使用 React/Vant 等框架默认开启 XSS 防护；后端返回的 JSON 内容均经过转义，若需渲染 HTML 则进行白名单过滤。  
   - CSRF 防御：后端对所有状态修改接口（POST/PUT/DELETE）强制要求 `X-CSRF-Token` 令牌，前端从 `/api/csrf-token` 获取并随请求一起提交。  

5. **日志与审计**  
   
   - 关键操作（登录、工单状态变更、规则修改、数据导出、对象存储上传/下载）在操作日志表 (`system_logs`) 中记录用户、IP、时间、操作类型、前后状态。  
   - 日志采用异步写入（Kafka 或 Redis Stream）后存入 Elasticsearch，便于检索与告警。

### 9. 性能设计

1. **缓存策略**  
   
   - Redis 用于存储 JWT 黑名单、短信验证码、热点商品信息、频繁访问的知识库条目（LRU 策略、TTL 5~30 分钟）。  
   - 某些查询（如热销商品TOP10、近一天咨询量）使用 Redis 计数器或有序集合实时更新。  

2. **数据库读写分离**  
   
   - MySQL 主库负责写操作，从库承担读查询（通过中间件如 ProxySQL 或 Spring 的 `@Transactional(readOnly=true)`）。  
   - 对高并发的咨询查询与用户订单列表使用读从库。  

3. **异步处理**  
   
   - 耗时操作（情感批量分析、爬虫清洗、MD 生成、报表导出）全部交给 Celery Worker，使用 Redis 作为 broker，防止阻塞请求线程。  
   - 长耗时的第三方调用（如调用 LLM）采用超时重试机制（最多 2 次），并在出现超时时返回友好提示（“正在思考中，请稍等...”）而不占用连接过长时间。  

4. **分页与限制**  
   
   - 所有列表接口默认返回不超过 50 条，客户端可通过 `page`、`size` 参数自行控制，最大 `size` 不超过 200。  

5. **负载均衡与水平扩容**  
   
   - 业务服务层（FastAPI）部署多副本后使用 Nginx 或云负载均衡器（如 ALB）做轮询。  
   - AI 引擎层同样可以水平扩展，因为其服务基本无状态（仅依赖外部模型 API 与缓存）。  
   - 数据层（MySQL 主从、Redis Sentinel、Elasticsearch 集群、Milvus Standalone）均具备基本的高可用能力。  

6. **性能目标（参考概要说明书）**  
   
   - H5 首页加载 ≤ 3s（3G 网络）  
   - L1 级智能回复 latency ≤ 800ms（含网络）  
   - 工单查询接口响应 ≤ 300ms  
   - 单实例业务 QPS ≥ 500  
   - 日处理售后工单 ≥ 10 万条  
   - 日处理评价 ≥ 50 万条  
   - 异步任务评价情感批量分析（10 万条） ≤ 10 分钟  
   - 异步任务爬虫数据清洗（50 万条） ≤ 30 分钟

### 10. 错误处理与日志

1. **统一错误返回格式**  
   
   ```json
   {
     "code": <int>, // 业务错误码，0 表示成功
     "message": <string>, // 人 readable 描述
     "data": <object|null>, // 成功时返回的数据
     "traceId": <string> // 用于链路追踪的唯一 ID
   }
   ```
   
   常见错误码：  
   
   - 1001：参数校验失败  
   - 1002：未授权（token 失效或未提供）  
   - 1003：禁止访问（角色不足）  
   - 1004：资源不存在  
   - 1005：业务冲突（如重复提交售后申请）  
   - 1006：第三方服务调用失败  
   - 1007：系统内部错误  

2. **异常捕获**  
   
   - 在所有控制器方法外层使用统一拦截器（Spring 的 `@ControllerAdvice` 或 FastAPI 的依赖异常处理）捕获异常，记录详细堆栈，返回对应错误码。  

3. **日志策略**  
   
   - 服务启动、关闭、重要配置变更记录到 `operation_log` 表。  
   - 每个请求在入口中产生 traceId（如 UUID），并在日志中输出，便于在 Elasticsearch 中做链路追踪。  
   - 错误日志（级别 ERROR 及以上）实时写入 Elasticsearch 的 `error_log` 索引，并通过告警规则触发企业微信/钉钉通知。  
   - 性能日志（响应时间 > 阈值）同样写入专门索引 `slow_log`，供监控分析。
