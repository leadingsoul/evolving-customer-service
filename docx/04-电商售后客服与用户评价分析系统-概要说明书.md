## 总体描述

### 产品视野

本系统为电商售后客服与用户评价分析一体化平台，定位于为消费者、商家客服及平台管理员提供全链路服务。系统主要与以下外部系统或设备交互：

- 消费者端：通过微信小程序或H5页面访问，获取订单信息并进行售后咨询、申请及评价。
- 商家客服端：通过Web管理后台登录，处理用户咨询、售后工单及服务记录。
- 平台管理员端：通过Web管理后台登录，进行知识库管理、售后规则配置、客服管理及数据分析。
- 第三方服务：包括通义千问/DeepSeek等大模型API、短信服务（阿里云/腾讯云）、对象存储（OSS/Mino）、物流查询接口以及电商平台公开评价爬虫（淘宝、天猫等）。

> **系统交互视图说明**：系统由四层架构组成——接入层（CDN+Nginx）、用户端（H5/客服端/管理端）、业务服务层（FastAPI）、AI引擎层（FastAPI子应用）以及数据层（MySQL、Redis、ES、Milvus、对象存储）。各层通过内部RESTful或WebSocket接口通信，且数据层仅限内网访问，确保安全隔离。

### 产品功能

系统核心功能包括：

1. **消费者端功能**：一键登录、订单查询、智能咨询（L1/L2/L3分级）、售后申请（退换货、仅退款等）、售后进度查看、多维评价提交（文字+图片）、个人中心管理。
2. **商家客服端功能**：工作台监控、咨询会话处理（AI辅助回复、快捷回复、会话转接）、工单处理（状态流转、AI分类、问题标签）、售后分类管理、服务记录与数据导出、个人绩效统计。
3. **平台管理员端功能**：系统概览、商品知识库管理、售后规则配置、客服账号管理、工单统计、评价分析（情感趋势、主题归类）、公开评论爬虫管理、评价对策库管理、AI模型配置、Agent自进化管理（会话沉淀、进化效果报告）。
4. **AI智能引擎功能**：意图识别、多轮对话上下文管理、RAG智能回复生成、情感分析、主题归类、工单自动分类、评价对策匹配、会话自动沉淀与自进化、智能分流与分级处理。

### 用户特性

- **消费者**：主要使用移动设备（微信小程序/H5），对操作流程要求简单便捷，期望快速得到售后问题解答及售后状态通知。
- **商家客服**：多为在线客服人员，需高效处理大量咨询与工单，借助AI辅助提升响应速度与准确率，后台界面需清晰展示待办事项及绩效指标。
- **平台管理员**：负责系统全局配置与数据分析，具有一定的技术背景，需要强大的数据可视化与管理能力，能够进行规则调整、模型调优及内容审核。
- **系统管理员/运维人员**：负责系统部署、监控、备份及安全维护，要求系统具备良好的可观测性、可扩展性及容灾能力。

### 运行环境

- **硬件平台**：推荐采用容器化部署（Docker Compose 用于开发/测试，Kubernetes 用于生产）。生产环境最小集群规模包括：FastAPI Pod×3、Celery Worker Pod×2、AI引擎 Pod×2、MySQL主从、Redis Sentinel、Elasticsearch集群、Milvus Standalone、MinIO双节点或云OSS bucket。
- **软件平台**：后端采用 Python 3.11+、FastAPI、SQLAlchemy、Celery 等；前端使用 Vite + TypeScript + Vant 4（H5）、Vite + React 18 + Ant Design 5（客服端/管理端）；数据库为 MySQL 8.0+、Redis 7.0+、Elasticsearch 8.x、Milvus 2.3+；对象存储支持 MinIO 或 阿里云 OSS；消息队列使用 Redis Broker。
- **网络环境**：系统采用前后端分离架构，内部服务通过内部网络通信；外部访问仅暴露 80/443 端口（HTTPS），所有对外接口强制 TLS ≥ 1.2；数据层（MySQL、Redis、ES、Milvus、对象存储）仅允许内网访问，禁止公网直连。

### 设计约束

- 必须采用微信一键登录方式，支持手机号绑定及电商平台账号关联。
- 所有敏感数据（手机号、订单信息）落库需加密存储（AES-256），日志中脱敏处理。
- 文件上传限制为 jpg、png、pdf，单文件 ≤ 5MB，需病毒扫描。
- 系统需支持水平扩展，业务服务层及AI引擎层均为无状态设计，可通过增加Pod数量实现扩容。
- 必须提供完整的监控、日志及告警机制（Sentry+Prometheus、ELK/OpenTelemetry）。
- 系统需符合等保2.0基本要求，具备身份鉴权、访问控制、安全审计及防护能力。
- 接口采用RESTful或WebSocket，前后端交换数据使用JSON格式。

### 假设与依赖

- 假设所选用的LLM API（通义千问/DeepSeek）具备稳定的服务可用性及足够的调用额度。
- 假设第三方短信服务（阿里云/腾讯云）可正常发送售后进度通知。
- 假设电商平台公开评价爬虫目标站点（淘宝、天猫）允许在遵守robots.txt及平台合规条款下进行数据采集。
- 假设系统运维团队具备容器化及Kubernetes基础运维能力。
- 依赖的基础软件（MySQL、Redis、ES、Milvus等）版本满足上述最低要求。

## 系统结构

系统采用分层微服务架构，主要包括以下五层：

1. **接入层**：负责静态资源分发、SSL终结及流量分发，使用CDN加速静态资源，Nginx 作为反向代理。
2. **用户端层**：分别提供 H5 移动端（消费者入口）、React SPA 客服端（商家客服入口）及 React SPA 管理端（平台管理员入口）。
3. **业务服务层**：基于 FastAPI 实现的无状态后端服务，提供RESTful API 支持用户、工单、评价、客服等业务模块，同时提供WebSocket支持实时消息推送。
4. **AI引擎层**：作为独立的FastAPI子应用或独立服务，封装意图识别、多轮对话、RAG回复生成、情感分析、主题归类、工单自动分类等AI能力，支持Horizontal Pod Autoscaler。
5. **数据层**：内部存储与检索系统，包括：
   - MySQL 8.0 主从复制，存储业务核心数据（用户、订单、工单等）。
   - Redis 7.0+ Sentinel，用于缓存、会话存储及Celery消息队列Broker。
   - Elasticsearch 8.x 集群，用于评价全文检索与日志检索。
   - Milvus 2.3+ 向量数据库，用于商品知识库语义检索及FAQ相似匹配。
   - MinIO或阿里云OSS，用于存储凭证图片、会话日志、MD沉淀文档及数据备份。

> **分层架构图说明**：系统自上而下为接入层 → 用户端层 → 业务服务层 → AI引擎层 → 数据层。接入层通过CDN将静态资源（H5/客服端/管理端前端文件）分发至用户终端；Nginx进行SSL终结后将请求转发至业务服务层（/api/*）或前端静态资源；业务服务层根据路由分发至对应业务模块，同时调用AI引擎层获取智能服务；AI引擎层从数据层（MySQL、Redis、 Milvus、OSS）读取知识、模型及缓存数据；数据层仅内网可达，确保数据安全。

## 运行需求

### 性能需求

- **页面加载时长**：H5首页及登录页在3G网络下≤3秒，客服端及管理端首屏渲染≤2秒（在典型办业务网络下）。
- **响应时长**：普通咨询（L1级）AI回复延迟≤800ms；工单查询、状态变更等业务接口响应≤300ms；复杂查询（如评价分析、工单统计）响应≤1.5秒。
- **并发支持**：系统需支持峰值并发用户数2000（消费者端），同时在线客服数200，管理端并发用户数50；业务服务层每实例应能够处理每秒500请求（QPS）。
- **吞吐量**：单日处理售后工单量≥10万条，评价提交量≥50万条，咨询会话量≥30万次。
- **异步任务处理**：评价情感批量分析任务（单批10万条）完成时间≤10分钟；爬虫数据清洗任务（单日采集50万条评论）完成时间≤30分钟。

### 安全需求

- 所有敏感信息传输须使用HTTPS（TLS≥1.2），且采用HSTS防止降级攻击。
- 密钥管理：所有API Key、数据库密码、JWT Secret等通过环境变量或K8s Secret注入，禁止硬编码或提交到代码仓库。
- 输入校验：所有前端提交及后端接口均需进行严格的输入合法性校验（长度、类型、正则等），防止SQL注入、XSS及CSRF攻击。
- 访问控制：不同角色（消费者、客服、管理员）仅能访问其授权的接口及数据；后台管理接口需额外添加验证码及登录失败次数限制。
- 日志审计：关键操作（登录、工单状态变更、规则修改、数据导出等）必须记录详细审计日志，便于事后追溯。

### 可靠性需求

- 业务服务层及AI引擎层采用无状态设计，支持水平扩展与滚动更新，单节点故障不影响整体服务可用性。
- 数据库采用主从复制（MySQL）或集群模式（ES、Milvus），具备自动故障转移能力。
- Redis采用Sentinel哨兵模式，提供高可用缓存与消息队列服务。
- 对象存储采用MinIO分布式模式或云OSS多区域副本，确保数据持久性。
- 系统需提供自动备份与恢复机制：MySQL每日全量+ binlog增量备份，保留30天；Redis AOF+RDB混合持久化并每日快照；Elasticsearch每日快照仓库备份；对象存储支持跨区域复制。
- 必须配置健康检查接口（如`/api/health`），用于容器编排的就绪与存活探测。

### 可维护性需求

- 代码结构遵循领域驱动设计（DDD），按业务域划分模块，便于定位与维护。
- 所有依赖项版本均须锁定，并通过`requirements.txt`或`package.json`进行管理，确保构建可重现。
- 提供完整的API文档（使用Swagger/OpenAPI），并在内部网络可访问；Swagger UI在生产环境默认关闭或仅内网IP可访问。
- 构建流程采用自动化CI/CD，代码提交后自动执行Lint、单元测试、构建镜像并推送至制品仓库。
- 监控与告警：通过Prometheus采集服务指标，使用Grafana展示监控面板；通过Alertmanager将告警通知发送至企业微信/钉钉/邮件。
